# ARIA Knowledge Format v1.0 (Draft)

## 0. About this document

This is the normative specification for the **ARIA Knowledge format** — an open, MCP-native format for typed, validated, structured knowledge that AI agents and humans can both read, write, and reason over.

The format describes what makes a **Core** (a structured knowledge repository) conformant. It is independent of any single implementation. The `aria-core` project is the reference implementation; third parties are welcome and encouraged to build their own.

**License:** Apache 2.0. Implementers are granted a perpetual, worldwide, royalty-free patent license to use any claims embodied in this specification.

**Governance:** This spec is editor-led (Mike Prasad) for v1.0. A 3–5 person editorial body is planned for v2.0, drawn from RFC participants and active implementers.

**How to comply:** A Core implementation that claims ARIA Knowledge format v1.0 conformance MUST satisfy all MUST clauses below and SHOULD satisfy all SHOULD clauses unless documented otherwise in the implementation's conformance report.

## 1. Vision and motivation

The ARIA Knowledge format exists because **no open, MCP-native, agent-readable, validation-rich format for structured knowledge currently exists**. Adjacent registries (GitHub for code, Hugging Face for models, MCP Registry for servers) cover adjacent artifacts. None publishes a normative shape for *durable structured knowledge* — the typed, validated, rationale-anchored material that AI agents need at runtime and that humans need for institutional memory.

The format's core proposition: **a Core captures not only WHAT but WHY**. The WHY-axis — alternatives considered, criteria applied, causal mechanisms, validation outcomes — is what makes knowledge genuinely agent-reusable across time and context. Without it, an agent consuming a Decision can only blindly apply or ignore. With it, an agent can reason about whether the original criteria still apply under new conditions.

## 2. Goals and non-goals

### Goals (v1.0)

- Define a normative item schema, a normative knowledge-type vocabulary, and a normative archetype vocabulary for validation.
- Define a normative MCP tool surface (`core.*` namespace) that conformant Cores expose.
- Define normative composition primitives — **Shard import** (one-shot snapshot) and **Exocore subscription** (live-pull from another Core).
- Define the universal base fields every knowledge item must carry, drawn from W3C PROV-O and the IEEE Std 1012 verification framework.
- Define eight validation modes mapped to archetype families.
- Define optional semantic retrieval interface for implementations that choose to support it.
- Allow implementations to choose their own storage backend, transport hardening, and access controls — the spec governs the wire shape, not the implementation.

### Non-goals (v1.0)

- The spec does not mandate a specific embedding model, vector index, or semantic-search algorithm.
- The spec does not mandate trust score formulas, audit certificate formats, or registry curation policies. Those are platform concerns (see aria-synapse).
- The spec does not specify UI conventions. Visual, conversational, and programmatic clients may render Cores however they wish.
- The spec does not enforce a fork/PR workflow. v1.0 uses citation, extension, and derivation; fork primitives are deferred to a future spec version.

## 3. Conformance levels

A Core implementation declares one of the following conformance levels in its `core.json` (see §10):

| Level | What it means |
|-------|---------------|
| **`core-base`** | Implements universal base fields, knowledge types, archetypes, item schema, the 13 normative MCP tools, Shard import/export, Exocore subscribe (read-only). MUST clauses only. |
| **`core-extended`** | All of `core-base`, plus Exocore subscribe (write-back), optional `core.validate` event sourcing, optional semantic retrieval interface. SHOULD clauses also satisfied. |
| **`core-experimental`** | Declares pre-v1.0 features (additional MCP tools, additional archetypes, custom extensions). Useful for RFC-window implementations; not v1.0-conformant. |

The Synapse platform (aria-synapse) requires `core-extended` for inclusion in its registry. Self-hosted Cores not joining the registry may declare any level.

## 4. Universal base fields

Every knowledge item, regardless of type or archetype, MUST carry the following fields. The base layer is drawn from W3C PROV-O and the epistemic-AI literature on knowledge provenance.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | UUID v4 string | yes | Globally unique item identifier |
| `type` | enum | yes | One of the nine user-facing types in §6 |
| `archetype` | List[archetype enum] | yes | One or more archetypes from §5 (minimum 1; multi-valued allowed) |
| `content` | string (Markdown) | yes | The knowledge statement itself |
| `source` | string | yes | Origin entity — person, system, document, sensor, or inference engine |
| `source_type` | enum | yes | One of `observed`, `learned`, `told`, `inferred`, `computed`, `experienced` |
| `timestamp_created` | ISO 8601 datetime | yes | When the knowledge was first acquired |
| `timestamp_valid_from` | ISO 8601 datetime | recommended | When the knowledge began to be true in the world (may equal `timestamp_created`) |
| `timestamp_valid_until` | ISO 8601 datetime, nullable | optional | Expiry or invalidation time. `null` means open-ended |
| `confidence` | float 0.0–1.0 | recommended | Publisher's certainty in the knowledge |
| `scope` | string | yes | Domain or context of applicability (e.g., `software_engineering`, `securities_law`, `pediatric_oncology`) |
| `language` | enum | yes | One of `natural_language`, `formal_logic`, `code`, `math`, `mixed` |
| `version` | string (semver-style) or integer | yes | For knowledge that evolves; defaults to `1` for items never edited |
| `lineage` | List[item-id] | optional | Chain of transformations or inferences that produced this item |
| `status` | enum | yes | One of `intake`, `promoted`, `deprecated`, `archived` |
| `tags` | List[string] | yes | Structured labels (may be empty array) |
| `extensions` | object | yes | Type- and archetype-specific extension fields (see §8). May be empty object only for `Reference` type |
| `provenance` | object | conditional | Required when item arrived via Shard import or Exocore subscription. See §15 |
| `validations` | List[ValidationEvent] | optional | Array of validation events (see §16) |
| `pii_check` | object | optional | PII scan status (see §16) |

**Implementation note:** Items SHOULD be stored in a way that preserves field ordering as listed above for cross-implementation diff stability. JSON-LD or canonical JSON output is recommended for export and Shard generation.

### 4.1 source_type semantics

The `source_type` field captures the *epistemic origin* of the knowledge — how the publisher came to know it. Use the value that best matches:

- `observed` — empirically witnessed or measured by the publisher
- `learned` — acquired through study, instruction, or training
- `told` — communicated by another agent or source the publisher trusts
- `inferred` — derived from other knowledge by reasoning
- `computed` — produced by a deterministic or algorithmic process
- `experienced` — acquired through episodic encounter or practice

A single item carries exactly one `source_type`. If an item could plausibly be tagged multiple ways, prefer the most direct origin.

## 5. Archetype vocabulary (31)

The `archetype` field is multi-valued and drawn from a controlled vocabulary of 31 archetypes. Each archetype captures a distinct *cognitive shape* of knowledge with its own validation strategy.

The vocabulary is presented in §8 alongside each archetype's extension fields and anchor requirements. The list of archetype enum values:

```
propositional, procedural, acquaintance, a_priori, a_posteriori, semantic,
episodic, implicit, schematic, script, factual, conceptual, metacognitive,
conditional, explicit, tacit, causal, relational, deductive, inductive,
abductive, analogical, heuristic, structural_meta, normative, causal_structural,
intuitive, embodied, distributed, strategic, generative
```

An item MUST declare at least one archetype. Most items carry 1–3. Implementations MAY validate that declared archetypes are consistent with the item's `type` (see §6 mapping) but MUST NOT reject items merely because the type/archetype pairing is unusual — domain authors are the authority on their own knowledge's shape.

**Why a multi-valued list:** real knowledge spans cognitive shapes. A Decision that records "we chose Postgres over MongoDB because relational joins on customer→subscription→invoice degrade at scale" is *simultaneously* reasoning-shaped (alternatives weighed) AND causal-shaped (cause/effect/mechanism) AND heuristic-shaped (success rate, failure conditions). Forcing one archetype loses validation fidelity.

## 6. User-facing types (9)

The `type` field is single-valued and drawn from a controlled vocabulary of 9 user-facing types. Types are the friendly labels publishers think in; archetypes are the underlying validation routing.

| Type | Primary archetypes | What it captures |
|------|-------------------|------------------|
| `decision` | reasoning, causal | A chosen course among alternatives, with rationale and (ideally) cause–effect |
| `approach` | procedural, heuristic, conditional | A way to do something under specified conditions |
| `lesson` | episodic, abductive | What happened, what was expected, why outcomes diverged |
| `risk` | causal, heuristic | A potential harm with likelihood, impact, mitigation |
| `hypothesis` | abductive, inductive | A predicted outcome with falsifiability test |
| `fact` | propositional + (a_posteriori OR semantic) | An asserted truth — observed empirically OR known generally |
| `definition` | a_priori, factual, conceptual | An accepted truth within a domain (formal, axiomatic, or terminological). Consolidates Mechanism, Formula, Principle, Theorem, Term, Theory |
| `scenario` | schematic, script | A defined situation referenced by other items via `applies_when` |
| `reference` | explicit, relational | A pointer to external authoritative source. Not rationale-gated; gated for citation aliveness only |

**Note on `decision` ≠ `reasoning`:** the type is a publisher-facing label; the archetype is the validation shape. Publishers who think they're writing "decisions" tag them as `type: decision`; the format then validates against the declared archetypes — which usually include `reasoning` and `causal` but MAY include `heuristic` or others depending on the item.


## 8. Per-archetype extension fields and anchor requirements

For each archetype an item declares, **at least one of the archetype's anchor fields** MUST be populated for the item to pass the promotion gate (see §9). Implementations MAY declare additional extension fields beyond those listed here.

This section lists the 31 archetypes with their canonical extension fields and the anchor subset (bold in the field column).

## 9. Promotion gate (anchor-based)

For an item to transition from `status: intake` to `status: promoted`, the implementation MUST verify the **per-archetype anchor gate**:

> For each archetype declared in the item's `archetype` list, at least one of that archetype's anchor fields (the **bold** rows in §8) MUST be populated (non-empty, non-null, non-empty-collection).

Items declaring multiple archetypes must satisfy the anchor requirement *for each archetype declared*. This is the format's encoding of the "knowledge captures WHY, not just WHAT" principle.

`Reference` typed items are exempt from the anchor gate. They are gated for citation aliveness instead (see §16).

### 9.1 Reference algorithm

```python
def passes_anchor_gate(item):
    if item.type == "reference":
        return True, "reference items are hygiene-gated, not anchor-gated"
    
    if not item.archetype:
        return False, "item must declare ≥1 archetype"
    
    ext = item.extensions or {}
    populated = {k for k, v in ext.items() if v not in (None, "", [], {})}
    
    for archetype in item.archetype:
        anchors = ARCHETYPE_ANCHORS[archetype]  # from §8 bold rows
        if not (populated & anchors):
            return False, f"archetype '{archetype}' needs ≥1 anchor: {sorted(anchors)}"
    
    return True, "OK"
```

The promotion gate is normative for conformant implementations. Registries (e.g., Synapse) MAY impose additional gates beyond this minimum.

## 12. MCP tool surface (`core.*` namespace) — 14 tools

A conformant Core MUST expose the following 14 tools over MCP. Implementations MAY expose additional tools but MUST NOT use the `core.*` namespace for non-spec tools.

### 12.1 Read (4 tools, anyone with valid token)

| Tool | Purpose |
|------|---------|
| `core.search` | Full-text + optional semantic + filters. Args: `query`, `mode?`, `types?`, `archetypes?`, `tags?`, `status?`, `limit?` |
| `core.get` | Fetch one full item by id or slug |
| `core.list` | Browse without query. Args: `type?`, `archetype?`, `tag?`, `since?`, `limit?` |
| `core.recent` | Freshest items first. Args: `type?`, `limit?` |

### 12.2 Write (3 tools, anyone)

| Tool | Purpose |
|------|---------|
| `core.store` | Create new intake item (`status: intake`) |
| `core.update` | Edit own drafts (or any if owner) |
| `core.propose_promotion` | Mark intake item ready for owner review |

### 12.3 Curate (1 tool with action arg, owner only)

| Tool | Actions |
|------|---------|
| `core.curate` | `{ action: "approve" \| "reject" \| "archive", item_id, reason? }` — owner-only lifecycle transitions |

### 12.4 Shards (2 tools)

| Tool | Purpose |
|------|---------|
| `core.shard.export` | Package a versioned snapshot. Args: `version`, `filter?` |
| `core.shard.import` | Ingest a `.shard` with provenance preserved. Args: `shard_payload` or `shard_url` |

### 12.5 Exocores (1 tool with action arg)

| Tool | Actions |
|------|---------|
| `core.exocore` | `{ action: "connect" \| "disconnect" \| "list" \| "fetch" \| "apply" \| "diff" \| "stash" \| "resolve" \| "detach", exocore_id?, source_core_url?, filter?, schedule?, merge_policy?, item_id? }` |

### 12.6 Validate (1 tool)

| Tool | Purpose |
|------|---------|
| `core.validate` | Record a validation event on an item. Args: `item_id`, `outcome` (`validated` \| `invalidated` \| `inconclusive`), `context` |

### 12.7 Relate (1 tool with action arg)

| Tool | Actions |
|------|---------|
| `core.relate` | `{ action: "link" \| "tag", subject_id, ... }` — typed links between items + tag operations |

`link` actions support typed relations: `supports`, `contradicts`, `refines`, `cites`, `applies_scenario`. Used (among other things) when an item's `applies_when` field references a `Scenario` item.

### 12.8 Summarize (1 tool)

| Tool | Purpose |
|------|---------|
| `core.summarize({ scope, query?, item_id?, max_tokens? })` | Return a summary of an item, a query result set, or the whole Core. Implementations use LLM where available; otherwise return first-N-chars heuristic. Distinct tool because summarization is a transform on content, not a relation operation. |

### 12.9 Total

4 Read + 3 Write + 1 Curate + 2 Shard + 1 Exocore + 1 Validate + 1 Relate + 1 Summarize = **14 tools.** Action-arg collapse on `core.curate`, `core.exocore`, `core.relate` is the spec's preferred shape for sub-action operations; standalone tools for `core.summarize`, `core.validate`, `core.search`, etc. where the operation has a single conceptual shape.

## 13. Composition primitives

A Core MAY intake knowledge from another Core via two primitives:

### 13.1 Shard import (one-shot snapshot)

A **Shard** is a versioned, frozen export of a Core or filtered subset. Wire format: `.tar.gz` with the following internal structure:

```
shard.manifest.json    Shard metadata (source Core, version, item count, signature)
items/                 One JSON file per item (matching the §7 schema)
provenance.json        Original lineage chain at export time
LICENSE                Copy of source Core's license
README.md              Shard-level description
```

Shard import is invoked via `core.shard.import`. Items arrive with full provenance preserved (see §15).

### 13.2 Exocore subscription (live pull)

An **Exocore** is another live Core that the local Core subscribes to. Subscriptions support git-style fetch/apply/stash/resolve/detach semantics:

| Action | Effect |
|--------|--------|
| `connect` | Create subscription with filter + schedule + merge policy. First fetch happens immediately. |
| `disconnect` | End subscription. Imported items remain (provenance preserved). |
| `list` | Enumerate active subscriptions |
| `fetch` | Pull upstream delta to staging (no apply yet) |
| `apply` | Apply staged delta per merge policy |
| `diff` | Show what would change if apply ran |
| `stash` | Set aside local edits before apply |
| `resolve` | Resolve a conflicted item (pick local / upstream / manual) |
| `detach` | Stop syncing this exocore's items; keep them in place |

**Default merge policy: `preserve-local`** — local edits are never silently overwritten. Other options: `replace`, `fork-on-edit`, `manual`.

### 13.3 Subscription record schema

```jsonc
"subscriptions": [{
  "id": "uuid",
  "source_core_url": "https://core.example.com",
  "source_core_id": "uuid",
  "filter": { "types": ["decision", "approach"], "archetypes": ["causal"], "tags": ["startup-ops"] },
  "pull_schedule": "daily",
  "last_pulled_at": "2026-05-19T12:00:00Z",
  "lockfile_url": "https://core.example.com/lockfile?at=...",
  "merge_policy": "preserve-local"
}]
```

`lockfile_url` records exactly what items were pulled at last subscription update — same primitive as `package-lock.json` or `Cargo.lock`.

### 13.4 Per-item divergence states (subscribed items)

| State | Meaning |
|-------|---------|
| `clean` | No local edits since import; safe to overwrite on sync |
| `dirty` | Local edits exist; sync defers per merge policy |
| `conflicted` | Local edits AND upstream changed; requires owner resolution |
| `detached` | Owner explicitly forked off; no longer syncs from upstream |

## 14. Provenance and attribution

Every item that arrived via Shard import or Exocore subscription MUST carry provenance that survives indefinitely:

```jsonc
"provenance": {
  "source_core_id": "uuid-of-source-core",
  "source_core_name": "expert-legal/securities-precedent",
  "source_item_id": "uuid-of-original-item",
  "intake_method": "shard_import" | "exocore_subscription",
  "intake_at": "2026-05-19T12:00:00Z",
  "source_version": "v2.3.1",
  "chain": [/* previous provenance entries if item was re-imported */]
}
```

`chain` preserves lineage across multiple re-imports — critical for trust score attribution, citation, and validation telemetry.

## 15. Validation events (`core.validate`)

The `core.validate` tool records that an item has been validated in real-world use:

```jsonc
"validations": [{
  "id": "uuid",
  "outcome": "validated" | "invalidated" | "inconclusive",
  "context": "Applied to S-series acquisition in 2026-Q2",
  "validated_by": "user-id",
  "validated_at": "2026-05-19T12:00:00Z",
  "evidence_uri": "optional reference"
}]
```

Validation events are append-only. Aggregate validation outcomes feed trust scoring (a platform concern; see aria-synapse).

## 16. Validation modes (eight, mapped to archetypes)

The format recognizes eight validation modes, each appropriate for different archetype families. Conformant implementations SHOULD support consistency checking and provenance tracing at minimum; other modes are recommended where the implementation hosts the relevant archetypes.

| Validation mode | Applicable archetypes | Primary method |
|----------------|----------------------|----------------|
| Formal verification | a_priori, deductive, structural_meta | Theorem provers, OWL reasoners, SAT solvers |
| Empirical benchmark | a_posteriori, inductive, semantic, factual | Precision/recall vs. ground-truth dataset |
| Behavioral probe | procedural, implicit, intuitive, embodied | Test cases, adversarial inputs, execution trials |
| Expert review | tacit, normative, heuristic, causal | Domain specialist inspection |
| Provenance tracing | episodic, explicit, relational, distributed | Lineage chain, source freshness, PROV-O |
| Consistency checking | all (first-pass filter) | Automated contradiction detection — should run on all inserts |
| Abductive evidence | abductive, causal, analogical | Text evidence corpus matching, explanation pattern recall |
| Calibration testing | metacognitive, strategic, conditional | Confidence vs. accuracy correlation on known-answer tasks |

Implementations SHOULD declare which validation modes they support in `core.json` `capabilities`. Registries MAY require a minimum set for inclusion.

### 16.1 PII check (deterministic hygiene)

Every item MAY carry a `pii_check` block recording the most recent PII scan:

```jsonc
"pii_check": {
  "last_scanned": "2026-05-19T12:00:00Z",
  "status": "clean" | "flagged" | "redacted" | "not_scanned",
  "flagged_categories": ["email", "phone", "name"],
  "scan_version": "presidio-2.x"
}
```

PII scanning is a hygiene-level Integrity check (see aria-synapse for the platform's Integrity axis). For self-hosted Cores, PII handling is the implementation's choice.

## 17. Optional semantic retrieval interface

Implementations MAY support semantic retrieval. Format-level conformance does NOT require it.

### 17.1 Item-level fields (optional)

```jsonc
{
  "embedding": [0.013, -0.042, ...],
  "embedding_metadata": {
    "model": "text-embedding-3-small",
    "dimensions": 1536,
    "generated_at": "2026-05-19T12:00:00Z",
    "normalization": "l2"
  }
}
```

### 17.2 `core.search` `mode` arg

| Mode | Behavior |
|------|----------|
| `fts` | Full-text search only (default — universally supported) |
| `semantic` | Semantic similarity only |
| `hybrid` | Both, ranked combined |

Implementations that don't support `semantic` or `hybrid` SHOULD return a structured `not_supported` error rather than silently degrading. Consumers can use this to probe capability.

### 17.3 Interface convention

Implementations that support semantic mode SHOULD accept OpenAI-compatible embedding payloads — i.e., the request shape used by `POST /v1/embeddings` on OpenAI-compatible APIs. This is an *interface convention*, not a model mandate. Publishers choose model + dimensions and declare in `embedding_metadata`.

## 18. Versioning and ratification

The format follows semantic versioning:

- **Patch (1.0.x)**: clarifications, typo fixes, examples added. No schema changes.
- **Minor (1.1, 1.2, ...)**: additive optional fields and tools. Backward compatible.
- **Major (2.0)**: breaking changes to required fields or tools. Stated deprecation period.

### 18.1 Ratification flow

1. v0.1 RFC draft published on `spec.ariaknowledge.ai` and `github.com/mikeprasad/aria-knowledge-spec`
2. Community feedback window — open for at least 60 days from v0.1 publication
3. RFC consolidation pass — editor reviews feedback, decides type vocabulary, archetype vocabulary, and load-bearing schema concerns
4. v1.0 candidate published with all changes integrated
5. v1.0 ratified after 30-day quiet period with no breaking issues raised

### 18.2 Conformance test suite

A reference conformance test suite ships alongside v1.0 in the spec repo. Test artifacts:

- Sample valid Cores at `tests/fixtures/cores/`
- Sample valid Shards at `tests/fixtures/shards/`
- Schema validators for `core.json`, item schema, Shard manifest
- MCP-tool contract tests (request/response shape verification for all 13 tools)
- Anchor-gate test cases (positive + negative for each archetype)
- Provenance-chain reconstruction tests

Any implementation may claim v1.0 conformance by running the suite against itself and publishing the results.

## 19. Out of scope (v1.0)

The following are explicitly out of scope for v1.0. Some may land in v1.x or v2.0 based on RFC and adoption signal:

- Fork primitives (full Core clone with divergence)
- PR-style contribution workflows (propose-changes-back-to-source)
- Federation (runtime query fan-out across many Cores)
- Audit certificate format (PII, bias, poisoning — Synapse defines its own; not spec'd)
- Trust score formula (platform concern; not spec'd)
- Cross-Core semantic federation (per-Core semantic is in scope; cross-Core is v2+)
- Mass deletion (knowledge is archived, never deleted; physical deletion requires direct DB access)
- Vector embeddings as required (optional per §17)
- UI / rendering conventions

## Appendix C — Foundational references

This spec inherits from:

1. **IEEE Std 1012** — System, Software, and Hardware Verification and Validation. The four-mode model (knowledge base verification, domain validation, procedural validation, procedural optimization) anchors §16.
2. **W3C PROV-O** — Provenance Ontology. `lineage`, `source`, `source_type`, and `provenance.chain` derive from PROV-O.
3. **"Knowledge Archetypes for AI Systems: Taxonomy with Validation Data Requirements"** (ARIA Knowledge research, 2026) — Source of the 31-archetype vocabulary and per-archetype validation fields in §5 and §8.
4. **MPAI-MMC V2.5 Theory of Mind data model** — Influences the universal base layer pattern (`confidence`, `scope`, `version`, `lineage`).
5. **Model Context Protocol (MCP)** — Transport layer for the `core.*` tool surface in §12. See https://modelcontextprotocol.io.

## Appendix D — Open questions for the v0.1 RFC

The following are intentional unknowns to be resolved during the RFC window:

1. **Archetype vocabulary stability.** v1.0 ships all 31 archetypes. Should v2.0 trim, expand, or sub-classify based on adoption data?
2. **Anchor field set per archetype.** The bold rows in §8 reflect editor judgment. RFC may surface better anchor choices.
3. **Type vocabulary trim.** 9 user-facing types feels right; RFC should validate against publisher needs.
4. **Multi-archetype validation logic.** Currently each archetype's anchor must pass independently. Should there be aggregate scoring (e.g., a "rationale density" metric across archetypes)?
5. **Validation event aggregation.** Should validation outcomes be normalized to a `[0,1]` confidence score, or kept categorical?
6. **Reference type subtype.** Should `reference` split into `reference_paper`, `reference_url`, `reference_internal`?
7. **`scenario` as a referenced type.** When item A's `applies_when` field cites a Scenario item B, should the format require a typed relation rather than a string match?
